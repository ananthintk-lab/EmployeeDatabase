## Codebase Patterns
- Package root is `com.employee.EmployeeDatabase`; use sub-packages `model`, `repository`, `exception`, `controller`, `dto`, `advice` under it as features are added.
- Repository pattern: define a plain interface (e.g. `EmployeeRepository`) and back it with an `InMemoryEmployeeRepository` `@Repository` using `ConcurrentHashMap<Long, Employee>` + `AtomicLong` for thread-safe auto-increment IDs. No JPA/database — this project is intentionally in-memory only per CLAUDE.md requirements.
- `findAll()` must return a list sorted by id ascending (acceptance criteria across stories expect deterministic ordering) — sort in the repository, not the controller.
- `existsByEmail` does a case-insensitive comparison (`equalsIgnoreCase`) since email uniqueness should not be case-sensitive.
- Validation annotations (`@NotBlank`, `@Email`) go directly on the `Employee` model fields (jakarta.validation.constraints), consistent with spring-boot-starter-validation.
- On Windows, avoid PowerShell `-replace` with Get-Content/Set-Content on files containing non-ASCII characters (em-dashes etc.) — it mangled UTF-8 into mojibake in prd.json once. Prefer the Edit tool for any file with non-ASCII content.
- `mvnw test`/`verify` print a benign "Mockito is currently self-attaching" / byte-buddy-agent warning on this JDK — it is not a build failure, ignore it when scanning output for real errors; check `target/surefire-reports/*.txt` for actual pass/fail counts instead of relying on console noise.
- `@SpringBootTest` test classes share one cached Spring context (and one `InMemoryEmployeeRepository` instance) across the whole test run. For endpoints whose acceptance criteria require exact list-content assertions (e.g. "empty store", "N employees"), use `@WebMvcTest(EmployeeController.class)` + `@MockitoBean EmployeeService` instead so each test fully controls the data returned — see `EmployeeGetAllControllerTest`. Use `@MockitoBean` (not the deprecated `@MockBean`) on Spring Boot 3.4+.
- When manually verifying a running instance on Windows via `Start-Process ./mvnw.cmd spring-boot:run`, the returned `Start-Process` PID is the wrapper process, not the JVM on port 8080 — get the real PID via `Get-NetTCPConnection -LocalPort 8080 | Select-Object -ExpandProperty OwningProcess` and stop that one, or the port/log files stay locked after cleanup.
